🏗️ Porquê Microserviços?

Decisão arquitetural: Microserviços vs Monolito

💡 Decisão de Design

Microserviços foram escolhidos porque o projeto exige duas linguagens (C# + Python), e cada serviço tem requisitos distintos de performance e bibliotecas.

VantagemExemplo no Projeto
Separação de responsabilidadesSensor só publica, Análise só calcula
Tecnologias mistasC# para rede, Python para NumPy/SciPy
Isolamento de falhasSe Análise cair, Servidor continua
Deploy independenteAtualizar Python sem recompilar C#
Testabilidade14 testes C# + 25 testes Python

Porquê NÃO Monolito?

  • Não permite C# + Python na mesma app
  • Falha na Análise derrubaria todo o sistema
  • Escalar = duplicar tudo (desperdício)

Porquê NÃO Kubernetes?

  • Exagerado para 6 serviços académicos
  • Kubernetes exige clusters, pods, ingress
  • Docker Compose resolve com 1 ficheiro

📮 RabbitMQ & Protocolo AMQP

Comunicação assíncrona Sensor → Gateway via message broker

💡 Decisão: Porquê Topic Exchange?

Topic exchanges permitem routing flexível com wildcards. O Gateway subscreve sensor.*.# e recebe todas as medições de qualquer sensor, sem precisar de saber quantos existem.

Routing Keys // Exchange: sensor-measurements (Topic, durable) sensor.sensor-01.temperatura → medição específica zona.A.humidade → por zona geográfica // Gateway subscreve com wildcards: sensor.*.# → TODAS as medições zona.*.# → todas por zona

Código: Sensor publica no RabbitMQ

C# — RabbitMQSensorClient.cs var properties = new BasicProperties { ContentType = "application/json", DeliveryMode = DeliveryModes.Persistent, // disco! Timestamp = new AmqpTimestamp(...) }; await _channel.BasicPublishAsync( exchange: "sensor-measurements", routingKey: $"sensor.{SensorId}.{tipoDado}", mandatory: false, basicProperties: properties, body: body // JSON UTF-8 );

Decisões Chave:

  • DeliveryMode Persistent — medições sobrevivem a reinicios
  • autoAck: false — ACK manual após processar
  • Heartbeat: 30s — keep-alive da ligação
  • AutomaticRecovery: true — reconexão automática

🌐 Gateway — Consumer & Orquestrador

Consome do RabbitMQ, invoca PreProcessamento (RPC), envia ao Servidor (TCP)

Código: Declarar Queues e Bindings

C# — RabbitMQGatewayClient.cs // Queue com nome único por gateway await _channel.QueueDeclareAsync( queue: $"gateway-measurements-{GatewayId}", durable: true, exclusive: false, autoDelete: false ); // Bind: receber TODAS as medições await _channel.QueueBindAsync( queue: $"gateway-measurements-{GatewayId}", exchange: "sensor-measurements", routingKey: "sensor.*.#" // wildcard! );

Código: ACK manual após processar

C# — RabbitMQGatewayClient.cs // Consumidor com autoAck: false (seguro) await _channel.BasicConsumeAsync( queue: $"gateway-measurements-{GatewayId}", autoAck: false, // ACK manual! consumer: consumer1 ); // Só fazemos ACK após processamento await _channel!.BasicAckAsync( queued.DeliveryTag, multiple: false );
💡 Decisão: ConcurrentQueue interna

As mensagens do RabbitMQ são colocadas numa ConcurrentQueue<QueuedMessage> interna. Um worker separado processa sequencialmente, evitando race conditions no envio TCP.

C# — Processamento sequencial private async Task ProcessarFilaAsync(CancellationToken ct) { while (!ct.IsCancellationRequested) { if (_messageQueue.TryDequeue(out var queued)) { await OnMensagemRecebida.Invoke( this, queued.Message); await _channel!.BasicAckAsync( queued.DeliveryTag, multiple: false); } else await Task.Delay(100, ct); } }

🔌 Sockets TCP — Protocolo Customizado

Comunicação Gateway → Servidor via TCP com protocolo JSON+newline

💡 Decisão: TCP puro vs HTTP

HTTP adiciona ~200 bytes de headers por medição. Com payloads de ~150 bytes, os headers representariam >50% do tráfego. TCP puro elimina esse overhead com parsing trivial: ReadLine() até ao \n.

Protocolo definido:

Protocolo TCP // Formato: JSON + newline (\n) // Encoding: UTF-8 // Tamanho máximo: 10 MB por linha // Gateway → Servidor: {"tipo":"DATA","sensor_id":"sensor-01", "payload":{"tipo_dado":"temperatura","valor":23.5}, "timestamp":"2025-06-05T10:30:00Z"}\n // Servidor → Gateway: {"tipo":"DATA_ACK","sensor_id":"sensor-01", "payload":{},"timestamp":"..."}\n

Código: Servidor TCP multi-threaded

C# — Servidor/Program.cs TcpListener listener = new(IPAddress.Any, listenPort); listener.Start(); while (true) { TcpClient client = listener.AcceptTcpClient(); gatewayCountMutex.WaitOne(); // thread-safe gatewayCount++; int num = gatewayCount; gatewayCountMutex.ReleaseMutex(); // Thread dedicada por Gateway Thread t = new(() => HandleGateway(client, num)) { IsBackground = true }; t.Start(); }

📞 RPC — Remote Procedure Call

Gateway → PreProcessamento (Python) · Servidor → Análise (Python)

💡 Decisão: HTTP/JSON vs gRPC

Escolhemos HTTP/JSON pela simplicidade: trivial em C# e Python, debugável com curl, sem geração de código .proto. gRPC seria evolução futura para escala.

Código: RPC Client com Retry

C# — PreProcessamentoClient.cs private static async Task<T?> ComRetryAsync<T>( Func<Task<T?>> action, int maxRetries = 2) { for (int i = 0; ; i++) { try { return await action(); } catch (Exception ex) when (i < maxRetries) { // Backoff exponencial: 1s, 2s, 3s... await Task.Delay(1000 * (i + 1)); } catch { return null; } } }

Código: Servidor Python (Análise)

Python — Analise/servico.py class RPCHandler(BaseHTTPRequestHandler): def do_POST(self): path = urlparse(self.path).path if path == "/rpc/estatisticas": self._estatisticas(dados) elif path == "/rpc/padroes": self._padroes(dados) elif path == "/rpc/previsao": self._previsao(dados)

💡 Decisão: SharedProtocol como .dll

Em vez de copiar classes entre projetos, o SharedProtocol é um assembly partilhado. Uma alteração no formato da mensagem propaga automaticamente para Sensor, Gateway e Servidor — princípio DRY.

💾 Persistência — SQLite

Base de dados embebida no Servidor com thread-safety via lock

💡 Decisão: SQLite vs PostgreSQL/MongoDB

SQLite é embutido (sem container extra), zero configuração, e suporta 100K+ writes/segundo. Com 6 sensores (~1000 medições/hora), não há necessidade de um SGBD dedicado.

Código: Schema da BD

C# — ServidorMonitor.cs string createSql = @" CREATE TABLE IF NOT EXISTS medicoes ( id INTEGER PRIMARY KEY AUTOINCREMENT, sensor_id TEXT NOT NULL, tipo_dado TEXT NOT NULL, valor REAL NOT NULL, timestamp TEXT NOT NULL, payload_json TEXT );"; string createAnalises = @" CREATE TABLE IF NOT EXISTS analises ( id INTEGER PRIMARY KEY AUTOINCREMENT, sensor_id TEXT NOT NULL, tipo_dado TEXT NOT NULL, tipo_analise TEXT NOT NULL, resultado TEXT NOT NULL, timestamp TEXT NOT NULL );";

Código: Persistir medição (thread-safe)

C# — ServidorMonitor.cs public bool PersistirMedicao(...) { lock (_dbLock) // thread-safe! { using var conn = new SqliteConnection(_cs); conn.Open(); using var cmd = new SqliteCommand(@" INSERT INTO medicoes (sensor_id, tipo_dado, valor, timestamp) VALUES ($sid, $td, $val, $ts);", conn); cmd.Parameters.AddWithValue("$sid", sensorId); cmd.Parameters.AddWithValue("$td", tipoDado); cmd.Parameters.AddWithValue("$val", valor); cmd.Parameters.AddWithValue("$ts", timestamp); cmd.ExecuteNonQuery(); } }
💡 Concorrência

Mutex no Servidor para gatewayCount e gatewayMap. lock no ServidorMonitor para escrita na BD. Garante integridade ACID mesmo com múltiplas Gateways simultâneas.

🐳 Docker Compose — Orquestração

6 serviços, rede isolada, healthchecks, dependências automáticas

Código: docker-compose.yml (excertos)

YAML — docker-compose.yml gateway: depends_on: rabbitmq: condition: service_healthy # espera healthcheck! servidor: condition: service_started pre-processamento: condition: service_started environment: - SERVER_ENDPOINT=servidor:7000 - RABBITMQ_HOST=rabbitmq # nome DNS interno - PRE_PROCESSAMENTO_HOST=pre-processamento healthcheck: test: ["CMD", "sh", "-c", "grep -q Gateway /proc/1/cmdline"] interval: 30s

Código: Healthchecks

YAML — Healthchecks por serviço # RabbitMQ test: ["CMD", "rabbitmq-diagnostics", "-q", "ping"] # Python (PreProc, Análise, Interface) test: ["CMD", "python", "-c", "import urllib.request; print( urllib.request.urlopen( 'http://127.0.0.1:5001/health').status)"] # C# (Gateway, Servidor) test: ["CMD", "sh", "-c", "grep -q Gateway /proc/1/cmdline"]
💡 Decisão chave

O Gateway tem condition: service_healthy no RabbitMQ — só arranca quando o broker está realmente funcional, não apenas "a correr". Isto evita erros de ligação no arranque.

🔄 Fluxo de Dados Completo

Da medição do sensor até ao dashboard web — passo a passo

1
Sensor publica medição no RabbitMQ (exchange topic, routing key: sensor.{id}.{tipo})
2
RabbitMQ encaminha via routing key match (sensor.*.#) para a queue do Gateway
3
Gateway invoca Pré-Processamento via RPC HTTP → uniformizar + validar (com retry + backoff)
4
Gateway envia dados tratados ao Servidor via TCP socket (JSON+newline). Aguarda DATA_ACK.
5
Servidor persiste em SQLite (tabela medicoes) + invoca Análise via RPC se necessário
6
Interface Web (porta 8000) consulta SQLite e mostra dashboard em tempo real com polling
3 Paradigmas no mesmo fluxo

Pub/Sub (Sensor→RabbitMQ) · RPC HTTP (Gateway→PreProc, Servidor→Análise) · TCP Sockets (Gateway→Servidor)

TCP Watchdog — Reconexão Automática

Lógica do Watchdog (30 segundos) WatchdogWorker(): 1. Verificar sensores inativos (>60s) 2. Atualizar CSV se houver alterações 3. Verificar ligação ao Servidor Se desconectado → ReconectarAsync() - 20 tentativas máximo - 5 segundos entre tentativas - Sucesso → reiniciar ciclo

💡 Isolamento de Falhas

Se a Análise (Python) falhar com divisão por zero, o Gateway continua a receber dados, o Servidor continua a persistir, e a Interface continua a mostrar o dashboard. O utilizador nem repara.
