
1. Levantamento de Requisitos (Requirements)
Requisitos Gerais do Sistema:
 * O sistema distribuído deve simular uma infraestrutura de monitorização ambiental urbana.
 * A arquitetura é composta por três tipos de entidades: SENSOR, GATEWAY e SERVIDOR.
 * A implementação de toda a comunicação deve ser feita usando sockets na linguagem C#.
 * O sistema tem de suportar atendimento concorrente através da utilização de threads.
 * Deve ser garantido o acesso sequencial aos ficheiros através da utilização de mutexes.
Requisitos do SENSOR:
 * O SENSOR é o dispositivo encarregue da recolha de dados ambientais.
 * Tem de suportar a recolha de dados como: temperatura, humidade, qualidade do ar, ruído, partículas (PM2.5/PM10), luminosidade e imagem/vídeo.
 * Na sua inicialização, o SENSOR precisa de receber o endereço IP da GATEWAY a que se vai ligar como parâmetro.
 * O dispositivo tem de se identificar com o seu ID, indicar os tipos de dados que recolhe e enviar medições.
 * Deve enviar mensagens de heartbeat periodicamente para indicar que continua ativo.
 * É necessário implementar uma interface de texto simples para o utilizador simular a recolha e o envio de dados.
Requisitos da GATEWAY:
 * A GATEWAY é a entidade intermédia que agrega e encaminha os dados dos sensores.
 * Deve manter ficheiros de configuração CSV (formato: sensor_id:estado:zona:[tipos_dados]:last_sync) para gerir os sensores associados.
 * A entidade precisa de validar se os sensores estão registados e qual o seu estado (ativo, manutenção ou desativado).
 * Tem de monitorizar os heartbeats e atualizar o campo last_sync para identificar sensores que deixaram de comunicar.
 * A GATEWAY tem de ser capaz de responder e processar pedidos de múltiplos sensores em simultâneo.
Requisitos do SERVIDOR:
 * O SERVIDOR é a entidade que armazena e analisa a informação recolhida.
 * Os dados recebidos devem ser armazenados em ficheiros distintos, que estarão organizados por tipo de dados.
 * A entidade tem de receber pedidos de múltiplos gateways simultaneamente de forma consistente.
 * Como funcionalidade extra, a informação recebida pode ser armazenada numa base de dados relacional.
2. Histórias de Utilizador (User Stories)
Com base nestes requisitos, podemos extrair as principais histórias para guiar o desenvolvimento por fases:
 * Como SENSOR, quero receber o IP da Gateway ao arrancar e estabelecer ligação, para poder registar a minha presença na rede.
 * Como SENSOR, quero enviar um heartbeat periódico, para que a Gateway saiba que continuo operacional e não me marque como indisponível.
 * Como SENSOR, quero utilizar uma interface de texto, para que eu (o aluno/utilizador) possa simular a criação de uma medição ambiental específica.
 * Como GATEWAY, quero ler um ficheiro CSV quando um sensor se liga, para validar se ele está registado e se o seu estado é "ativo".
 * Como GATEWAY, quero processar os dados recebidos de vários sensores ao mesmo tempo usando threads, para não bloquear a comunicação com outros sensores.
 * Como GATEWAY, quero encaminhar os dados validados para o Servidor correto, para que a informação seja centralizada.
 * Como SERVIDOR, quero receber dados de várias Gateways em simultâneo e usar mutexes, para garantir que não corrompo os ficheiros de texto ao gravar os dados por tipo.
