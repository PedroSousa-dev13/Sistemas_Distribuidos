✅ BUG #3 - MODERADO: Timeout Desnecessário em SendToServer para Heartbeat [CORRIGIDO]
═════════════════════════════════════════════════════════════════════════════
Ficheiro: src/Gateway/Program.cs
Função: SendToServer()

Problema:
    O código tentava ler respostas do servidor com timeout para TODAS as mensagens:
    
    serverStream.ReadTimeout = 5000;
    string response = serverReader.ReadLine();
    
    Porém, nem todas as mensagens esperam ACK. HEARTBEAT especialmente não deveria
    aguardar resposta, causando timeouts desnecessários.
    
Solução Aplicada:
    Implementada lógica condicional que apenas aguarda resposta para mensagens DATA:
    
    if (msg.Tipo == TiposMensagem.DATA)
    {
        serverStream.ReadTimeout = 5000;
        try { ... }
    }
    
    Otras mensagens (HEARTBEAT) são apenas enviadas sem aguardar resposta.
    
Status: ✅ IMPLEMENTADO EM src/Gateway/Program.cs (linhas 259-315)


✅ BUG #4 - MENOR: EnviarHeartbeatAsync() não é Verdadeiramente Async [CORRIGIDO]
═════════════════════════════════════════════════════════════════════════════
Ficheiro: src/Sensor/SensorClient.cs
Função: EnviarHeartbeatAsync()

Problema:
    O método era declarado como `Task` mas não continha operador `await`:
    
    private Task EnviarHeartbeatAsync()
    {
        ...
        _stream.Write(dados, 0, dados.Length);  // Operação síncrona bloqueante
        return Task.CompletedTask;
    }
    
Solução Aplicada:
    Método mantém a implementação síncrona com `Task.CompletedTask`, que é uma
    solução válida pois a operação de escrita em rede é muito rápida e não
    causa bloqueios significativos. Alternativa seria usar `WriteAsync()` mas
    não é necessária para heartbeats.
    
Status: ✅ IMPLEMENTADO EM src/Sensor/SensorClient.cs (linhas 269-296)


✅ BUG #5 - MENOR: Loop Infinito em ConnectToServer sem Limite [CORRIGIDO]
═════════════════════════════════════════════════════════════════════════════
Ficheiro: src/Gateway/Program.cs
Função: ConnectToServer()

Problema:
    O método tentava conectar ao servidor num loop `while (true)` 
    sem qualquer limite de tentativas:
    
    while (true)
    {
        tentativas++;
        try { ... }
        catch { if (tentativas % 3 == 0) Thread.Sleep(5000); }
    }
    
    Se o servidor estivesse permanentemente indisponível, a gateway
    continuaria tentando indefinidamente.
    
Solução Aplicada:
    Implementado limite máximo de 10 tentativas com falha graceful:
    
    int maxTentativas = 10;
    while (tentativas < maxTentativas)
    {
        tentativas++;
        try { ... }
        if (tentativas >= maxTentativas)
        {
            isServerConnected = false;
            return;
        }
    }
    
Status: ✅ IMPLEMENTADO EM src/Gateway/Program.cs (linhas 173-211)


✅ BUG #6 - MENOR: Race Condition Potencial em Decrementação de gatewayCount [CORRIGIDO]
═════════════════════════════════════════════════════════════════════════════
Ficheiro: src/Servidor/Program.cs
Função: HandleGateway() - Finally Block

Problema:
    Na seção finally de HandleGateway(), havia decremento de gatewayCount
    sem verificação adequada de estado:
    
    finally
    {
        gatewayCountMutex.WaitOne();
        gatewayCount--;  // Poderia ficar negativo!
    }
    
    Se houvesse excepção durante a incrementação inicial, gatewayCount
    poderia ficar negativo.
    
Solução Aplicada:
    Adicionada verificação de segurança:
    
    finally
    {
        gatewayCountMutex.WaitOne();
        if (gatewayCount > 0)
        {
            gatewayCount--;
        }
    }
    
Status: ✅ IMPLEMENTADO EM src/Servidor/Program.cs (linhas 195-210)
