# RESUMO EXPLICATIVO DO PROJETO: SISTEMA DE MONITORIZAÇÃO AMBIENTAL URBANA

## VISÃO GERAL DO SISTEMA
Este é um sistema de monitorização ambiental urbana distribuído composto por três tipos de componentes que se comunicam através de um protocolo definido:
1. **Sensores**: Dispositivos que capturam dados ambientais (temperatura, humidade, qualidade do ar, etc.)
2. **Gateways**: Intermediários que agregam dados de múltiplos sensores e os encaminham para o servidor
3. **Servidor**: Sistema central que persiste, processa e disponibiliza os dados coletados

## CONCEITOS FUNDAMENTAIS DE MATERIA (PARA ENTENDER O PROJETO)

### O QUE É TCP/IP?
**TCP (Transmission Control Protocol)** é um protocolo de comunicação que estabelece uma conexão confiável entre dois pontos em uma rede. É como uma "ligação telefônica" onde:
- Dados são enviados em ordem garantida
- Se algum pacote se perder, é retransmitido automaticamente
- Não há duplicação ou perda de dados (desde que a conexão mantenha-se ativa)
- Funciona sobre **IP (Internet Protocol)** que trata do endereçamento e roteamento

No projeto:
- Sensores abrem conexões TCP com Gateways (IP:5000)
- Gateways abrem conexões TCP com Servidores (IP:6000)
- Cada conexão permanece aberta enquanto o componente estiver ativo

### O QUE SÃO THREADS (EXECUÇÃO CONCORRENTE)?
**Threads** são linhas de execução independentes dentro de um mesmo programa que podem running em paralelo. Pense em ter várias pessoas fazendo tarefas diferentes simultaneamente no mesmo escritório.

No projeto:
- **Thread principal**: Gerencia a lógica principal do componente (Sensor/Gateway/Servidor)
- **Thread de Heartbeat** (no Sensor): Executa em segundo plano enviando sinais de vida a cada 20 segundos sem bloquear o envio de medições
- **Threads de I/O**: O .NET usa internamente threads para operações de rede não-bloqueantes (leitura/escrita em sockets)

Isso permite que um sensor:
- Envie uma medição
- Enquanto aguarda resposta, continue processando
- Independente disso, envie heartbeats a cada 20 segundos
- Tudo sem que uma operação bloqueie as outras

### O QUE É SERIALIZAÇÃO/DESSERIALIZAÇÃO?
Antes de enviar dados pela rede, objetos da memória precisam ser convertidos em formato que possa viajar (geralmente bytes). Depois, do lado receptor, esse formato precisa ser convertido de volta em objetos utilizáveis.

No projeto:
- **Serialização**: Objeto `Mensagem` → String JSON → Bytes UTF-8 → Enviado via TCP
- **Desserialização**: Bytes recebidos → String JSON → Objeto `Mensagem`
- Usamos `System.Text.Json` que é eficiente e built-in no .NET

### O QUE É PROTOCOLO DE APLICAÇÃO?
É um conjunto de regras que define **como** os componentes devem se comunicar para realizar tarefas específicas. Enquanto TCP/IP garante que os bytes cheguem corretamente, o protocolo de aplicação define o **significado** desses bytes.

No projeto, nosso protocolo de aplicação define:
- Quais tipos de mensagens existem (REGISTER, DATA, etc.)
- Qual estrutura elas devem ter (tipo, sensor_id, payload, timestamp)
- Como componentes devem responder a cada tipo
- Quais são os fluxos de comunicação esperados

### O QUE É ARQUITETURA EM CAMADAS (LAYERED ARCHITECTURE)?
É organizar o sistema em níveis onde cada camada só conhece e interage com a camada imediatamente abaixo/acima. Cada camada tem uma responsabilidade bem definida.

No projeto vemos isso em:
- **Camada de Transporte**: TCP/IP (gerencia conexões de rede)
- **Camada de Protocolo**: Nossas mensagens JSON (define o "vocabulário" da comunicação)
- **Camada de Aplicação**: Lógica específica de sensores/gateways/servidores (o que fazer com as mensagens)
- **Camada de Persistência**: Arquivos .txt no servidor (onde os dados ficam armazenados)

### O QUE É EVENT-DRIVEN PROGRAMMING?
É um paradigma onde o fluxo do programa é determinado por eventos: mensagens de rede, timer expirando, ações do usuário, etc.

No projeto:
- Receber uma mensagem via TCP gera um evento "mensagem recebida"
- Um timer de 20 segundos gera um evento "enviar heartbeat"
- Conexão cair gera um evento "desconectado"
- Tratamos esses eventos com handlers/métodos específicos

### O QUE É BLOQUEIO VS NÃO-BLOQUEIO?
- **Bloqueio**: Uma operação que só retorna quando completa (ex: aguardar resposta indefinidamente)
- **Não-bloqueio**: Uma operação que retorna imediatamente, dizendo se pode continuar ou tentar mais tarde

No projeto usamos principalmente abordagens não-bloqueantes com timeouts:
- `ReceberMensagemAsync` usa CancellationTimeout para não travar para sempre
- Tentativas de conexão têm timeouts definidos
- Isso permite que o sistema continue respondendo mesmo quando há problemas de rede

## ARQUITETURA DE COMUNICAÇÃO (DETALHES)
Os componentes comunicam-se através de conexões TCP diretas:
- Sensores ↔ Gateway (porta 5000)
- Gateway ↔ Servidor (porta 6000)
Cada conexão é persistente enquanto o componente estiver ativo, permitindo comunicação bidirecional em tempo real.

## ESTRUTURA DAS MENSAGENS
Todas as mensagens seguem o mesmo formato JSON compacto:
```json
{
  "tipo": "STRING",           // Tipo da mensagem (REGISTER, DATA, etc.)
  "sensor_id": "STRING",      // ID único do sensor
  "payload": { },             // Dados específicos do tipo de mensagem
  "timestamp": "STRING"       // Marca temporal ISO 8601 em UTC
}
```
As mensagens são terminadas com \n para facilitar a leitura em streams TCP.

## FLUXO DE TRABALHO PRINCIPAL

### 1. Registo do Sensor
Quando um sensor inicia:
1. Establish connection TCP com o Gateway (IP:porta 5000)
2. Envia mensagem REGISTER com lista de tipos de dados que suporta
3. Aguarda resposta:
   - REGISTER_OK: Registo bem-sucedido, pode começar a enviar dados
   - REGISTER_ERR: Falha no registo (sensor não existe, etc.), encerra conexão

### 2. Envio de Dados Ambientais
Após registo bem-sucedido, o sensor pode:
1. Enviar medições periódicas usando mensagens DATA
2. Cada DATA contém: tipo_dado (ex: "temperatura") e valor medido
3. Aguarda DATA_ACK do Gateway como confirmação de receção
4. Se não receber ACK dentro do timeout, pode tentar reenviar

### 3. Manutenção da Conexão (Heartbeat)
Para garantir que a conexão permanece ativa:
1. Sensor envia HEARTBEAT a cada 20 segundos
2. Gateway responde com HEARTBEAT_ACK
3. Se o Gateway não receber 3 heartbeats consecutivos (60s), marca o sensor como requiring manutenção
4. Se o sensor não receber HEARTBEAT_ACK, pode tentar reconectar

### 4. Encaminhamento para o Servidor
O Gateway atua como ponte:
1. Recebe mensagens DATA dos sensores
2. Encaminha-as exatamente como recebidas para o Servidor (mesma porta 6000)
3. Aguarda DATA_ACK do Servidor
4. Envia DATA_ACK de volta ao sensor original
5. Este processo é transparente para o sensor

### 5. Tratamento de Erros
Qualquer componente pode enviar mensagens ERROR quando detecta problemas:
- INVALID_FORMAT: JSON malformado ou campos obrigatórios em falta
- SENSOR_NOT_FOUND: Tentativa de registo com ID inexistente
- SERVER_UNAVAILABLE: Servidor offline (enviado pelo Gateway)
- INVALID_DATA_TYPE: Tipo de dado não suportado na mensagem DATA

## PERSISTÊNCIA DE DADOS
No lado do Servidor:
1. Cada tipo de dado ambiental tem seu próprio arquivo de texto (ex: temperatura.txt, humidade.txt)
2. Formato de cada linha: `[timestamp] [sensor_id] [valor]`
3. Exemplo: `2024-01-15T10:35:00.000Z SENSOR_001 23.5`
4. Os arquivos são adicionados apenas (append-only) para simplicidade

## EXTENSIBILIDADE DO SISTEMA
O projeto foi diseñado para ser facilmente extensível:

**Para adicionar novos tipos de mensagem:**
1. Definir nova constante em `TiposMensagem.cs`
2. Adicionar ao conjunto `TiposMensagem.Validos`
3. Criar factory method opcional em `Mensagem.cs`
4. Documentar no PROTOCOLO.md (seção 11.1)
5. Implementar handlers nos componentes que processam o novo tipo

**Para adicionar novos tipos de dados ambientais:**
1. Documentar na seção 5 do PROTOCOLO.md (unidade, tipo do valor)
2. Nenhuma alteração de código necessária (payload é genérico)
3. Atualizar Servidor para persistir o novo tipo (criar novo arquivo .txt)

## FLUXO DE DADOS COMPLETO EXEMPLO
1. Sensor SENSOR_001 inicia e conecta ao Gateway em 192.168.1.100:5000
2. → Envia: `{"tipo":"REGISTER","sensor_id":"SENSOR_001","payload":{"tipos_dados":["temperatura","humidade"]},"timestamp":"2024-01-15T10:30:00.000Z"}`
3. ← Recebe: `{"tipo":"REGISTER_OK","sensor_id":"SENSOR_001","payload":{},"timestamp":"2024-01-15T10:30:01.000Z"}`
4. → Envia medição: `{"tipo":"DATA","sensor_id":"SENSOR_001","payload":{"tipo_dado":"temperatura","valor":23.5},"timestamp":"2024-01-15T10:35:00.000Z"}`
5. Gateway encaminha para Servidor em 192.168.1.200:6000
6. ← Servidor responde DATA_ACK ao Gateway
7. Gateway encaminha DATA_ACK para o Sensor
8. Sensor continua enviando medições e heartbeats a cada 20 segundos

- Execução concorrente através de threads
- Programação orientada a eventos
- Tratamento de operações de I/O não-bloqueantes
