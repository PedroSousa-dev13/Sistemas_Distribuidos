2025 Sistemas Distribuídos Trabalho prático nº 2 Serviços de Análise e Monitorização Urbana  para One Health   Recorde o primeiro trabalho prático desenvolvido. O sistema simula a infraestrutura de monitorização ambiental urbana composta por sensores, gateways e servidores, usando comunicação baseada em sockets e protocolos definidos manualmente. Pretende-se agora dar continuidade ao sistema desenvolvido, evoluindo-o para uma arquitetura distribuída mais robusta, mantendo as funcionalidades anteriores e introduzindo novos mecanismos de comunicação e processamento. Esta nova fase deverá contemplar: • A utilização de RPC (Remote Procedure Call) para ligação entre componentes, nomeadamente: o entre o Gateway e serviços externos de pré-processamento de dados  o entre o Servidor e servidores específicos de análise de dados,  • A substituição da comunicação direta entre Sensores e Gateways por um sistema Publish/Subscribe, em que os Sensores publicam os dados recolhidos e os Gateways subscrevem os tópicos correspondentes aos tipos de dados de interesse;  A seguinte figura apresenta uma visão geral do sistema.

    O serviço distribuído  Deverão ser implementados os seguintes componentes e procedimentos, considerando os seus requisitos funcionais e tecnológicos:  Comunicação RPC Devem ser implementados procedimentos remotos para processamento em dois pontos críticos: Gateway ↔ Serviço de Pré-processamento: O Gateway utiliza RPC para invocar um serviço externo que assegura a uniformização dos dados (ex: conversão de escalas, tratamento de formatos JSON/XML/CSV) antes da agregação.   Servidor ↔ Serviço de Análise e Previsão: O Servidor invoca, via RPC, um serviço especializado para realizar análises estatísticas, deteção de padrões de poluição ou previsão de riscos para a saúde pública.

Comunicação baseada em Publicação/Subscrição A ligação entre os Sensores  e os Gateways deve utilizar um sistema de publicação/subscrição (Pub/Sub), baseado em RabbitMQ.  Cada SENSOR publica os dados recolhidos (temperatura, humidade, PM2.5, etc.) em tópicos específicos.  Os tópicos podem ser organizados por tipo de dado ou por zona.  Os GATEWAYs subscrevem os tópicos de interesse das zonas que gerem e/ou tipos de dados que devem gerir e encaminhar, permitindo uma receção de dados desacoplada e escalável.  Funcionalidades adicionais O Servidor deverá armazenar os dados recebidos e os resultados das análises realizadas numa base de dados (relacional ou NoSQL), permitindo a persistência da informação para posterior consulta. O sistema deverá disponibilizar uma interface simples, acessível via linha de comandos ou interface web, permitindo a visualização dos resultados das análises realizadas. O pedido de novas análises pode também ser despoletado por esta interface, suportando a parametrização dos dados a analisar (ex: intervalo temporal, tipo de sensor, identificação do sensor). Além do processamento de dados em diferentes formatos, é também considerado fator de valorização do trabalho a inclusão de componentes do sistema implementados com recurso a diferentes tecnologias e linguagens de programação, e.g. na simulação dos dados recolhidos pelos sensores e/ou na implementação dos serviços RPC.
