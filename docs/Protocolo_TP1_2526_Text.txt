Licenciatura em Engenharia Informática UTAD | ECT | DE – Sistemas Distribuídos 2025/2026 Hugo Paredes | Tiago Pinto | Cristiano Pendão TP1 – Serviços de Monitorização Urbana para One Health  Sistemas Distribuídos Trabalho prático nº1 Serviços de Monitorização Urbana para One Health  O conceito One Health reconhece que a saúde humana, animal e ambiental estão fortemente interligadas. A monitorização de fatores ambientais urbanos — como qualidade do ar, níveis de ruído, temperatura, humidade ou poluição — pode fornecer indicadores relevantes para a identificação de riscos para a saúde pública. Em contexto urbano, sensores distribuídos pela cidade podem recolher dados ambientais continuamente. Estes dados, quando associados à localização e ao período temporal em que as pessoas estiveram expostas a determinados ambientes, podem revelar indícios relevantes para apoiar processos de diagnóstico ou investigação epidemiológica. Pretende-se criar um sistema distribuído que simule uma infraestrutura de monitorização ambiental urbana, composta por sensores que recolhem dados ambientais e os enviam para uma infraestrutura de agregação e processamento. A arquitetura do sistema é composta por três tipos de entidades: • SENSOR – dispositivo responsável pela recolha de dados ambientais; • GATEWAY – entidade intermédia responsável pela agregação e encaminhamento de dados; • SERVIDOR – entidade responsável pelo armazenamento e análise da informação recolhida. Os sensores encontram-se distribuídos por diferentes zonas da cidade e podem recolher diversos tipos de dados, tais como: temperatura; humidade; qualidade do ar; nível de ruído; concentração de partículas (PM2.5 / PM10); luminosidade; imagem/video. Os dados recolhidos podem posteriormente ser utilizados para identificar padrões ambientais associados a determinadas zonas e períodos temporais, podendo revelar correlações relevantes para estudos epidemiológicos ou investigação em saúde pública, enquadrados no paradigma One Health.  SENSOR O SENSOR é a entidade responsável pela recolha de dados ambientais no espaço urbano. Estes sensores encontram-se distribuídos por diferentes zonas da cidade, como parques, avenidas, zonas residenciais, áreas industriais ou zonas escolares. Cada sensor pode estar equipado com um ou vários dispositivos capazes de recolher diferentes tipos de informação ambiental, bem como imagens e video em tempo real. A informação recolhida por estes sensores permite caracterizar as condições ambientais de determinadas zonas da cidade. Por exemplo, sensores instalados numa determinada zona podem indicar:  timestamp:2026-03-10T09:15:00 sensor_id:S102 zona:ZONA_ESCOLAR

O SENSOR deve receber como parâmetro de inicialização o endereço IP do GATEWAY ao qual se encontra associado. Após iniciar, o SENSOR deve: • estabelecer ligação com o GATEWAY; • identificar-se através do seu ID de sensor; • indicar os tipos de dados ambientais que consegue recolher; • enviar medições ambientais recolhidas; • enviar necessidade de criação de streams de video; • enviar periodicamente uma mensagem de heartbeat, indicando que o sensor se encontra ativo; • terminar corretamente a comunicação. O heartbeat permite ao gateway verificar se o sensor continua disponível e operacional. Caso um sensor deixe de enviar heartbeat durante um determinado período de tempo, o gateway poderá assumir que o sensor se encontra indisponível. O SENSOR deve implementar uma interface de texto simples, permitindo ao utilizador simular a recolha e envio de diferentes tipos de medições ambientais.  GATEWAY O GATEWAY é responsável pela receção, validação e encaminhamento dos dados enviados pelos sensores urbanos. Cada gateway pode estar associado a um conjunto de sensores instalados numa determinada zona da cidade. O gateway funciona como uma entidade intermédia responsável por agregar os dados recolhidos pelos sensores e encaminhá-los para os servidores responsáveis pelo processamento da informação. Para tal, cada GATEWAY possui um conjunto de ficheiros de configuração em formato CSV, que contêm informação relativa aos sensores associados e aos tipos de dados suportados. Um ficheiro de configuração poderá conter informação no seguinte formato:  sensor_id:estado:zona:[tipos_dados]:last_sync Exemplo: S101:ativo:ZONA_CENTRO:[TEMP,HUM,RUIDO]:2026-03-10T08:45:00 S102:ativo:ZONA_ESCOLAR:[PM2.5,TEMP]:2026-03-10T09:00:00 S103:manutencao:ZONA_INDUSTRIAL:[AR,PM10]:2026-03-09T18:30:00  O estado de um sensor pode assumir os seguintes valores: • ativo – sensor em funcionamento normal • manutenção – sensor temporariamente indisponível • desativado – sensor removido ou desligado O GATEWAY deverá ser capaz de:

• verificar se o sensor que inicia comunicação está registado; • validar o estado do sensor; • receber medições ambientais provenientes dos sensores; • registar os dados recebidos; • atualizar os ficheiros de estado dos sensores; • monitorizar o envio de heartbeat dos sensores, atualizando o campo last_sync; • identificar sensores que deixaram de comunicar durante um período prolongado; • encaminhar os dados recebidos para os servidores responsáveis pelo processamento da informação. Após receber a medição, o gateway deverá validar se o sensor está registado, verificar se o tipo de dado é suportado, atualizar o estado do sensor e encaminhar os dados para o servidor responsável. O GATEWAY deve também ser capaz de responder a pedidos provenientes de vários sensores em simultâneo, garantindo a consistência da informação recebida mesmo quando vários sensores enviam dados em simultâneo.  SERVIDOR O SERVIDOR é responsável pelo armazenamento e processamento da informação ambiental proveniente dos diferentes gateways. Os dados recebidos representam medições ambientais recolhidas em diferentes zonas da cidade e em diferentes momentos temporais. Estes dados podem posteriormente ser utilizados para análise ambiental, investigação epidemiológica ou estudos relacionados com o impacto das condições ambientais na saúde das populações. Cada servidor pode processar um ou vários tipos de dados ambientais. Os dados recebidos poderão ser armazenados em ficheiros distintos, organizados por tipo de dados. O SERVIDOR deve ser capaz de receber pedidos provenientes de múltiplos gateways simultaneamente, garantindo a consistência da informação armazenada mesmo quando vários gateways enviam dados em simultâneo.  Protocolo O protocolo deve incluir as mensagens e os estados necessários para os diálogos SENSOR/GATEWAY/SERVIDOR necessários para implementar o serviço. Além das comunicações de início e finalização das comunicações, deverão ser consideradas as comunicações relativas à informação de ID da SENSOR, registo de tipo de dados ativos, e envio de dados.  Implementação A implementação deve ser efetuada usando sockets em C#;  O formato do relatório O trabalho deve ser acompanhado de um relatório, de até 3 páginas (excluindo anexos), descrevendo as opções de implementação e o protocolo definido para a comunicação SENSOR/GATEWAY/SERVIDOR. O código fonte desenvolvido deverá ser disponibilizado em repositório. Entrega e apresentação O trabalho deve ser submetido através do Moodle até ao final do dia 17 de abril. É obrigatória a apresentação do trabalho submetido na aula PL seguinte à data da entrega.
